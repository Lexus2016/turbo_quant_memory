# Turbo Quant Memory for AI Agents

![Заглавная иллюстрация Turbo Quant Memory](assets/readme-hero-ru.svg)

Другие языки: [English](README.md) | [Украинский](README.uk.md)

AI-инструменты для разработки быстрые. Их память нет.

Turbo Quant Memory даёт Claude Code, Codex, Cursor и другим MCP-клиентам устойчивый слой памяти, чтобы они перестали снова и снова платить один и тот же налог на контекст.

Если вы пользуетесь AI-агентами каждый день, память перестаёт быть приятным бонусом и становится частью базовой инфраструктуры.

## Почему это важно

Без слоя памяти команды снова и снова платят за одни и те же проблемы:

- агентам приходится перечитывать слишком много материалов, прежде чем начать действовать
- прошлые решения теряются между сессиями
- полезные выводы застревают в старых чатах
- каждая новая задача снова начинается со сборки контекста с нуля

Turbo Quant Memory превращает эти постоянные потери в переиспользуемую инфраструктуру.

## Что вы получаете

- Более компактный контекст по умолчанию: сначала короткий результат поиска, а полный контекст только по явному запросу
- Постоянную память проекта: решения, уроки, передачи контекста и повторяемые паттерны
- Переиспользуемые знания: можно продвигать проверенные заметки из проекта в глобальное пространство имён
- Локальный контроль: данные остаются на вашей машине в `~/.turbo-quant-memory/`
- Прозрачность для оператора: проверки здоровья, статистика хранилища, статус свежести и smoke-проверка
- Поддержку нескольких клиентов: один MCP-сервер для Claude Code, Codex, Cursor, OpenCode и других клиентов

## Коротко о главном

Turbo Quant Memory помогает AI-агентам помнить важное, по умолчанию забирать меньше контекста и переходить к полному контексту только тогда, когда задача действительно этого требует.

## Почему им продолжают пользоваться

- Он уменьшает повторную загрузку контекста.
- Он делает AI-сессии менее безличными и менее забывчивыми.
- Он даёт важным знаниям проекта постоянный дом вне истории чатов.
- Он делает retrieval управляемым, а не превращает каждую задачу в дамп целых файлов.
- Он даёт слой памяти, который можно проверить, протестировать и контролировать.

## Доказательство на этом репозитории

В репозитории лежат реальные результаты измерений в [benchmarks/latest.md](benchmarks/latest.md) и [benchmarks/latest.json](benchmarks/latest.json).

Текущий снимок измерений:

- Размер корпуса: 9 Markdown-файлов, 138 индексированных блоков
- Полная индексация: 4.00 с
- Пустой incremental index: 0.68 с
- Средняя задержка `semantic_search`: 75.14 мс
- Средняя задержка `hydrate`: 41.71 мс
- Средняя экономия по байтам для `semantic_search` без hydrate: 78.02%
- Средняя экономия по байтам для `semantic_search + hydrate(top1)`: 63.41%
- Средняя экономия по словам для `semantic_search` без hydrate: 83.84%
- Средняя экономия по словам для `semantic_search + hydrate(top1)`: 74.76%

Что это значит на практике:

- путь по умолчанию заметно легче, чем чтение полных файлов
- даже после `hydrate` по лучшему совпадению guided-путь всё равно остаётся существенно компактнее наивного сценария
- больше контекстного бюджета остаётся на рассуждение, а не на повторное чтение одних и тех же материалов

Метод измерения:

- Базовый сценарий без управляемой памяти: открыть полный текст каждого уникального Markdown-файла, который попал в top-5 результатов поиска по проекту
- Компактный путь: использовать только ответ `semantic_search`
- Guided-путь: использовать `semantic_search`, а затем `hydrate` для лучшего Markdown-результата

Это реальные числа для этого репозитория и этой реализации. Это не универсальная гарантия для любой кодовой базы.

## Установка

Рекомендуемая установка релизной версии из GitHub-тега:

```bash
uv tool install git+https://github.com/Lexus2016/turbo_quant_memory@v0.2.1
turbo-memory-mcp serve
```

Резервный путь через `pip`:

```bash
python -m pip install git+https://github.com/Lexus2016/turbo_quant_memory@v0.2.1
turbo-memory-mcp serve
```

Режим разработки из исходников:

```bash
uv sync
uv run turbo-memory-mcp serve
```

Редактируемая `pip`-установка из исходников:

```bash
python -m venv .venv
. .venv/bin/activate
pip install -e .
python -m turbo_memory_mcp serve
```

## Подключение клиента

Идентификатор сервера:

- `tqmemory`

Команда запуска:

- `turbo-memory-mcp serve`

Быстрые примеры:

- Claude Code: `claude mcp add --scope user tqmemory -- turbo-memory-mcp serve`
- Codex: `codex mcp add tqmemory -- turbo-memory-mcp serve`

После установки MCP-сервера и подключения его к клиенту память становится частью набора инструментов агента.

Не нужно запускать отдельные команды, открывать второе окно или вручную включать память для каждой задачи.

Дальше вы просто разговариваете с агентом обычным языком. Когда память полезна, агент может сам вызвать `tqmemory`.

Если хочется подчеркнуть это явно, можно написать "используй память проекта" или упомянуть `tqmemory`, но это необязательно.

## Что писать Codex

С MCP-сервером не нужно разговаривать shell-командами.

Вы пишете обычный текст в Codex, а Codex сам вызывает нужные инструменты памяти.

Главное: после подключения MCP не нужен специальный триггер для каждой задачи.

В обычной работе вы просто описываете задачу. Если память помогает, Codex должен использовать её автоматически.

Хорошие шаблоны запросов:

1. Первый запуск в новом репозитории

```text
Проиндексируй этот репозиторий и скажи, какая память теперь доступна для следующих задач.
```

2. Перед изменением кода

```text
Перед тем как что-то менять, проверь память проекта на прошлые решения по auth, sessions и retries, а потом коротко их суммируй.
```

3. Когда нужно найти правильный источник перед изменением

```text
Найди в проекте flow payment webhook, раскрой самый релевантный результат из памяти и объясни, как сейчас устроена реализация.
```

4. Когда нужно сохранить решение

```text
Сохрани решение в память как заметку с названием "Webhook retry policy" и кратким описанием подхода, о котором мы только что договорились.
```

5. Когда знание должно жить не только в этом проекте

```text
Если заметка, которую мы только что создали, пригодится и в других проектах, подними её в global memory.
```

6. Когда нужен полный сценарий за один запрос

```text
Найди всё важное про caching в этом репозитории, раскрой лучший результат и предложи изменение кода на основе этого контекста.
```

7. Когда хотите сказать явно, но всё равно по-человечески

```text
Сначала используй память проекта. Найди прошлые решения по caching и retries, а потом продолжай с изменением.
```

## Простой человеческий сценарий

- Один раз установите MCP-сервер и подключите его к агенту.
- После этого работайте с Codex как обычно. Вы описываете задачу, а память при необходимости подключается автоматически в фоне.
- В новом репозитории всё равно полезно один раз попросить проиндексировать материалы, чтобы памяти было что искать.
- Перед рискованными изменениями можно попросить агента сначала проверить память проекта.
- После важных решений стоит попросить агента сохранить их, чтобы следующие сессии не потеряли этот контекст.
- По умолчанию индексация проекта пропускает исторические и low-signal каталоги вроде `.planning`, `.serena` и сгенерированных benchmark-отчётов, чтобы активный поиск оставался сфокусированным на живых документах и заметках.

Самая простая ментальная модель такая:

- `semantic_search` — найти
- `hydrate` — раскрыть
- `remember_note` — сохранить
- `promote_note` — использовать позже в других задачах
- `deprecate_note` — убрать устаревшее знание из активного поиска без потери истории

## Когда знание устарело

- Сначала сохраните новую корректную информацию как новую заметку.
- Потом вызовите `deprecate_note` для старой заметки, если она больше не должна появляться в активном поиске.
- Если у старой заметки есть прямая замена, передайте id новой заметки, и старая будет помечена как `superseded`, а не просто архивирована.
- Так история не теряется, но `semantic_search` больше не поднимает старую инструкцию как актуальную.

## Технические детали

Модель пространств имён:

- `project`: локальные для репозитория заметки текущей кодовой базы
- `global`: переиспользуемые заметки, явно продвинутые из `project`
- `hybrid`: объединённый поиск по `project` и `global` с сильным приоритетом `project`

Порядок определения текущего проекта:

1. Нормализованный URL `origin`
2. Hash корневого пути репозитория, если remote отсутствует
3. Явные overrides через `TQMEMORY_PROJECT_ROOT`, `TQMEMORY_PROJECT_ID` и `TQMEMORY_PROJECT_NAME`

Набор инструментов:

- `health`
- `server_info`
- `list_scopes`
- `self_test`
- `remember_note`
- `promote_note`
- `deprecate_note`
- `semantic_search`
- `hydrate`
- `index_paths`

Где лежат данные:

- `~/.turbo-quant-memory/`

Команды проверки репозитория:

```bash
uv run pytest -q
uv run python scripts/smoke_test.py
uv run python scripts/benchmark_context_savings.py
```

## Ограничения и честные оговорки

- Проект не заявляет прямого контроля над KV-cache hosted-моделей.
- Первый embedding-backed запуск может скачать `sentence-transformers/all-MiniLM-L6-v2`, если локальный кеш холодный.
- Benchmark-отчёт измеряет этот репозиторий и эту реализацию, а не любой возможный сценарий внедрения.

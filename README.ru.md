Turbo Quant Memory — это слой памяти, который превращает AI-агента из “умного, но забывчивого помощника” в надёжного участника команды.

Если вы работаете с Claude Code, Codex, Cursor, OpenCode, Gemini CLI или любым MCP-клиентом, это способ сохранить и масштабировать знания команды между задачами.

## Почему Это Важно

Большинство AI-процессов ломаются в одном месте: память.

- Важные выводы теряются в истории чатов.
- Каждая новая задача стартует почти с нуля.
- Команда снова и снова объясняет одни и те же вещи.

Turbo Quant Memory делает знания постоянными, поисковыми и переиспользуемыми.

## Почему Команды Выбирают Turbo Quant Memory

| Типичный AI-процесс | С Turbo Quant Memory |
|---|---|
| Агент забывает контекст между сессиями | Агент продолжает работу с уже сохранённых знаний |
| Решения спрятаны в старых тредах | Решения становятся заметками, которые легко найти |
| Знания держатся на отдельных людях | Знания становятся общим активом команды |
| Контекст тратится на повторное чтение | Больше бюджета остаётся на рассуждение |

## Главная Обещанная Ценность

Ваши агенты перестают быть “временными чат-ассистентами” и начинают работать как полноценные члены команды.

## Что Делает Этот Проект Особенным

- Local-first подход: память остаётся под вашим контролем.
- Один слой памяти для многих клиентов.
- Непрерывность между агентами: можно начать в Codex, продолжить в Gemini CLI и вернуться в Codex с той же project memory.
- Фокус на реальной разработке: решения, паттерны, handoff-и.
- Прозрачность: знания структурированы и проверяемы.

## Быстрый Старт

Используйте этот сценарий на 60 секунд:

1. Установите один раз:
```bash
uv tool install git+https://github.com/Lexus2016/turbo_quant_memory@v0.4.2
```

2. Добавьте MCP-сервер `tqmemory` в клиент (клиент будет запускать его автоматически):

```bash
# Codex
codex mcp add tqmemory -- turbo-memory-mcp serve

# Gemini CLI
gemini mcp add tqmemory turbo-memory-mcp serve

# Claude Code (project scope)
claude mcp add --scope project tqmemory -- turbo-memory-mcp serve
```

3. Перезапустите клиент и вызовите любой инструмент `tqmemory`.

Нужен готовый конфиг для Gemini CLI, Cursor, OpenCode или Antigravity? Возьмите его из [CLIENT_INTEGRATIONS.ru.md](CLIENT_INTEGRATIONS.ru.md).

### Обновление

Чтобы подтянуть новый релиз в уже установленное дерево, повторите команду установки с `--reinstall`:

```bash
uv tool install --reinstall git+https://github.com/Lexus2016/turbo_quant_memory@v0.4.2
```

Если `~/.gemini/settings.json` создан до v0.4.2, единоразово смержите в него этот блок, чтобы Gemini CLI начал читать `AGENTS.md` вместе с `GEMINI.md`:

```jsonc
{
  "context": { "fileName": ["AGENTS.md", "GEMINI.md"] }
}
```

## Исключение Файлов При Индексации

Создайте файл `.tqmemoryignore` в корне проекта, чтобы исключить директории или файлы из Markdown-индексации. Формат аналогичен `.gitignore` — один glob-паттерн на строку, `#` для комментариев.

```gitignore
# Пропустить дубли workspace-шаблонов
workspace-*

# Пропустить runtime-отчёты
data/reports/*.md

# Пропустить сгенерированный контент
output/
```

Ignore-файл подхватывается автоматически при вызове `index_paths(...)`. Паттерны сопоставляются как с именами директорий, так и с полными относительными путями. Поиск идёт вверх от индексируемого корня до `.tqmemoryignore` или границы `.git`.

Директории `node_modules`, `.git`, `__pycache__`, `dist` и `build` всегда игнорируются по умолчанию.

## Общая Память Между Агентами

В стандартной локальной установке это работает из коробки. Не нужны отдельный sync-сервис, export/import или отдельная memory-настройка под каждого агента.

Это общая локальная память, а не удалённая облачная синхронизация. Если Codex и Gemini CLI работают на одной машине и открывают один и тот же репозиторий, они автоматически используют один и тот же memory layer.

Чтобы одна и та же project memory была доступна в Codex, Gemini CLI и других MCP-клиентах:

1. Один раз установите `turbo-memory-mcp` на машине.
2. Добавьте один и тот же MCP-сервер `tqmemory` в каждый клиент, которым пользуетесь.
3. Открывайте один и тот же репозиторий в каждом клиенте.

Когда эти условия соблюдены, клиенты автоматически видят одну и ту же project memory. То есть можно начать работу в Codex, продолжить в Gemini CLI, а потом вернуться в Codex без повторного восстановления контекста.

Если какой-то клиент запускается не из корня репозитория, явно задайте `TQMEMORY_PROJECT_ROOT`, чтобы он определял тот же project identity.

## Преимущество, Подтверждённое Бенчмарками И Экономией Денег

![Сводка бенчмарков](benchmarks/summary-ru.svg)

На реальном корпусе этого репозитория компактный memory-путь показывает сильную экономию, которая напрямую снижает расходы на модель:

- только `semantic_search`: в среднем **-63.96% байтов** в модель
- `semantic_search + hydrate(top1)`: в среднем **-44.1% байтов**
- средняя задержка `semantic_search`: **68.13 мс**
- средняя задержка `hydrate`: **41.63 мс**

Почему это практическое преимущество:

- меньше повторного чтения означает меньше платных входных токенов
- более низкий токен-объём означает более низкую стоимость каждой задачи
- контекстный бюджет идёт на рассуждение, а не на повторную загрузку файлов

## Что Добавлено В v0.4.1

- Автоматический failover прокси-процессов при гибели primary. Раньше, если первый `turbo-memory-mcp` процесс (primary, держащий sentence-transformers модель и LanceDB-дескрипторы) завершался, остальные прокси в других MCP-клиентах теряли RPC-связь и `tqmemory` становился тихо недоступен для этих клиентов.
- Прокси теперь детектируют пропавший primary через `PrimaryUnreachable` при следующем RPC-вызове и прозрачно ре-бутстрапятся: один из уцелевших прокси атомарно забирает lockfile и продвигает себя в primary (стартует собственный `DaemonListener`), а каждый другой осиротевший прокси переподключается к новому primary. Перезапуск MCP-клиентов не нужен.
- Phase-aware обработка RPC-ошибок: ошибки фазы connect/send транслируются в `PrimaryUnreachable` (безопасно для повтора), а ошибки mid-call (send прошёл, recv упал) поднимаются в хост без изменений, чтобы non-idempotent инструменты вроде `remember_note` никогда не дублировались.
- Фикстура Gemini CLI теперь поставляется с `context.fileName: ["AGENTS.md", "GEMINI.md"]`, поэтому Gemini CLI подхватывает те же project-промпты из `AGENTS.md`, что и Codex, Cursor и остальные агенты, без необходимости держать дублирующий `GEMINI.md`.

## Что Добавлено В v0.4.0

- Транспорт через singleton-daemon: только один процесс `turbo-memory-mcp` на машине держит модель sentence-transformers и LanceDB-дескрипторы. Каждый следующий запуск MCP-клиента становится тонким stdio↔socket прокси, который пересылает вызовы инструментов главному.
- Кроссплатформенная координация: на Unix/macOS используется `AF_UNIX` сокет в системной tmp-директории (короткий путь, права 0600); на Windows — named pipe текущего пользователя. Аутентификация через 32-байтовый случайный authkey в `~/.turbo-quant-memory/.daemon.lock`.
- Lazy-импорты в `retrieval_index` означают, что прокси не платит ~470 MB за импорты PyTorch / LanceDB / PyArrow, если только пересылает RPC. Измеренная экономия: ~1 ГБ RSS для четырёх одновременных MCP-клиентов (primary 530 MB, прокси ~50 MB каждый против 437 MB каждый раньше).
- Существующее состояние на диске (JSON-заметки, LanceDB-таблицы, Markdown-блоки, manifests) не меняется и полностью обратно-совместимо. Escape hatch: установить `TQMEMORY_DAEMON_DISABLE=1`, чтобы вернуться в per-process режим.

## Что Добавлено В v0.3.1

- Задокументирован shared-memory handoff между Codex и Gemini CLI в README и client integration docs.
- Добавлена готовая Gemini CLI фикстура и smoke-check шаги для проверки одного и того же `tqmemory` сервера в разных клиентах.
- Явно уточнено, что shared memory означает локальную непрерывность работы на одной машине, а не удалённую облачную синхронизацию.

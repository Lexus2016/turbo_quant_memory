<objective>
Wire the Phase 9-01 foundation into the live server: register `Subsystem.SECRETS` with the migration runner, ship the v0->v1 migration that provisions an empty vault for every existing project, expose four new MCP tools, and harden the indexer/lint/retrieval paths to refuse `secrets/` directory traversal.
</objective>

<tasks>

1. Extend `Subsystem` enum in `migrations/registry.py` with `SECRETS = "secrets"`. Add `SECRETS_FORMAT_VERSION = 1` constant in `store.py`. Extend `latest_version` constant map in `registry.py` to include the new subsystem. Update the docstring comment in `Subsystem` class.

2. Implement migration in `migrations/upgrades.py`:
   - `@migration(Subsystem.SECRETS, from_version=0, to_version=1, description="provision empty per-project secret vaults on existing projects")`.
   - The upgrade function walks `<storage_root>/projects/*`, and for every existing project_id calls `SecretsStore(storage_root, project_id).provision()`. If `resolve_master_key` raises `MasterKeyUnavailable` (e.g., headless box without env var set yet), the migration creates the `secrets/` directory + a stub `meta.json` with `{"version": 1, "vault_initialized": false}` but skips `vault.tqv` creation; on first successful `set_secret` the vault is initialized. This keeps migration unconditionally green and lazy-completes vault on first real use.
   - The runner's `_read_current_version` reads `secrets/meta.json.version` (default 0 if file missing); `_bump_manifest` writes the new version. Per-project manifest, not a single global manifest — matches existing per-project layout for other subsystems.
   - Unit test: synthetic storage with 3 project dirs (one with env-passphrase set, two without), run runner, verify each project has `secrets/meta.json`; project with key also has `vault.tqv`; running again is no-op.

3. Add MCP tools in `server.py` and matching response models in `contracts.py`:
   - `set_secret(name: str, value: str) -> SetSecretResult` — resolves current `ProjectIdentity`, calls `SecretsStore(...).set(...)`, records audit, returns `{"status": "ok", "name": "...", "project_id": "..."}`. On `MasterKeyUnavailable`, returns `{"status": "error", "code": "master_key_unavailable", "setup_hint": "<multi-line>"}`.
   - `get_secret(name: str) -> GetSecretResult` — returns `{"status": "ok", "name": "...", "secret_value": "...", "project_id": "..."}` or `{"status": "missing", "name": "..."}`. Crucially, `secret_value` is a dedicated field; never embedded into a descriptive `summary` / `message` field.
   - `list_secrets() -> ListSecretsResult` — returns `{"status": "ok", "names": [...], "project_id": "..."}`. Never any values.
   - `delete_secret(name: str) -> DeleteSecretResult` — returns `{"status": "ok", "deleted": true/false, "name": "..."}`.
   - Tool count in `self_test()` and `server_info()` goes `14 -> 18`. Update smoke fixtures.

4. Harden indexer + lint + retrieval against `secrets/` directory:
   - In `ingestion.py`, add a hard exclusion: any path that resolves through `<storage_root>/projects/<*>/secrets/` is refused with a clear `ValueError("secrets/ paths are not indexable")`. The exclusion is also added to the path-walker default ignore list.
   - In `knowledge_lint.py`, the same exclusion at the path-walker stage.
   - In `retrieval.py` and any retrieval-index loader, never traverse `secrets/` subdirectory. Add a regression test that explicitly attempts to index a secret-containing path and verifies it is rejected.

5. Update `server_info()` payload — **active project only**, no cross-project fan-out:
   - Add a `secrets` block: `{"vault_initialized": bool, "secret_count": N, "audit_entries_estimate": M, "key_resolution_mode": "env|keyring_existing|keyring_bootstrapped|unavailable"}`. `N` = count of entries in the active project's `vault.tqv` (requires master key — if unavailable, report `null` with `key_resolution_mode: "unavailable"` rather than failing the whole `server_info` call); `M` = line count of the active project's `audit.jsonl`. Never any names, never any values.

6. Tests:
   - `tests/unit/test_secrets_migration.py` covers idempotency, multi-project provisioning, manifest bump.
   - `tests/integration/test_secrets_mcp_tools.py` covers each of the four MCP tools end-to-end with a stub `keyring` backend.
   - `tests/integration/test_secrets_isolation.py` plants a fake secret file under `projects/X/secrets/`, registers that path with `index_paths`, and asserts the call is refused; also asserts `semantic_search` with a query matching the secret value returns nothing.

</tasks>

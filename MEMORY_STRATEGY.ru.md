# Стратегия Памяти

Другие языки: [English](MEMORY_STRATEGY.md) | [Ukrainian](MEMORY_STRATEGY.uk.md)

## Цель

Turbo Quant Memory сохраняет знания проекта доступными для поиска без необходимости держать полный текст в живом prompt.

Идея простая:

1. записывать локально
2. возвращать компактно
3. открывать больше только тогда, когда задаче действительно нужен более широкий контекст

## Рабочая Модель

| Слой | Что это означает |
|---|---|
| Один локальный MCP-сервер | Один stdio MCP-сервер обслуживает память для всех поддерживаемых клиентов |
| Project-first хранение | Заметки по умолчанию принадлежат текущему репозиторию |
| Управляемое глобальное переиспользование | В `global` попадают только явно продвинутые знания |
| Compact-first retrieval | Агент сначала просит минимально полезный ответ, а не весь сырой контекст |

Отдельного memory engine под каждого агента нет. Клиенты используют один серверный контракт и те же правила хранения.

## Активные Пространства

| Пространство | Роль | По умолчанию |
|---|---|---|
| `project` | Локальные знания и заметки репозитория | запись по умолчанию |
| `global` | Переиспользуемые знания между проектами | только через promotion |
| `hybrid` | Объединённый режим чтения с сильным приоритетом `project` | чтение по умолчанию |

### `project`

- Хранит знания, привязанные к одному репозиторию.
- Держит решения, уроки, handoff-и и паттерны рядом с кодовой базой, которая их породила.
- Является безопасным выбором для ежедневной работы.

### `global`

- Хранит знания, полезные за пределами исходного репозитория.
- Наполняется только через явную promotion.
- Сохраняет provenance до проектной заметки, из которой выросло правило или паттерн.

### `hybrid`

- Объединяет результаты из `project` и `global`.
- Отдаёт приоритет `project`, когда релевантность близка.
- Позволяет переиспользовать проверенные паттерны без потери привязки к текущему репозиторию.

## Идентичность Проекта

Текущая идентичность проекта определяется в таком порядке:

1. нормализованный URL `origin`
2. fallback на hash корневого пути репозитория
3. явные overrides

Поддерживаемые overrides:

- `TQMEMORY_PROJECT_ROOT`
- `TQMEMORY_PROJECT_ID`
- `TQMEMORY_PROJECT_NAME`

Это позволяет держать project memory стабильной между сессиями и при этом сохраняет контроль для нестандартных способов запуска.

## Структура Хранения

Хранилище file-backed и local-first:

```text
~/.turbo-quant-memory/
  projects/
    <project_id>/
      manifest.json
      notes/
        <note_id>.json
  global/
    manifest.json
    notes/
      <note_id>.json
```

Заметки и manifest-файлы пишутся атомарно через временный файл и `os.replace(...)`.

## Политика Записи

| Действие | Результат |
|---|---|
| `remember_note(..., scope="project")` | сохраняет типизированную project-note |
| прямой запись в `global` | отклоняется |
| `promote_note(note_id)` | создаёт переиспользуемую global-копию с provenance |
| `deprecate_note(...)` | выводит устаревшее знание из активного обращения без удаления истории |

Так `global` остаётся небольшим, управляемым и защищённым от случайного смешивания знаний из разных проектов.

## Политика Поиска

`semantic_search` поддерживает `project`, `global` и `hybrid`.

`hybrid` используется по умолчанию и работает так:

1. объединяет кандидатов из `project` и `global`
2. добавляет сильный бонус для `project`
3. при близких совпадениях предпочитает Markdown-блоки заметкам
4. финально разрешает ничью по приоритету `project`, затем по более новому `updated_at`, затем по стабильной идентичности

По умолчанию поиск работает и по проиндексированным Markdown-блокам, и по persistent memory notes.

## Контракт Карточки Результата

Каждый результат сохраняет provenance в явном виде. Компактный envelope включает:

- `scope`
- `project_id`
- `project_name`
- `source_kind`
- `item_id`
- `block_id`, если hit пришёл из Markdown
- `source_path`
- `title`
- `heading_path`
- `updated_at`
- `score`
- `confidence`
- `confidence_state`
- `compressed_summary`
- `key_points`
- `can_hydrate`
- `note_kind`, если это note-hit
- `promoted_from`, если это promoted global note

Стандартный retrieval не возвращает сырые excerpt-ы или дампы целых файлов. Эта граница снижает расход токенов и переносит более полный контекст в явные hydration-вызовы.

## Стратегия Hydration

Hydration явный и ограниченный:

| Режим | Поведение |
|---|---|
| `default` | target item плюс небольшое локальное окружение |
| `related` | target item плюс более широкое, но всё ещё ограниченное окружение |

Правила:

- Markdown hydration остаётся в пределах файла.
- Note hydration возвращает полное тело заметки вместе с metadata.
- Агент должен hydrate-ить контекст только тогда, когда компактного ответа уже недостаточно.

## Promotion И Provenance

Promoted global notes сохраняют блок `promoted_from`, который указывает назад на:

- исходный scope
- `project_id` источника
- `project_name` источника
- исходный `note_id`
- исходный `source_path`

Так межпроектное переиспользование остаётся прослеживаемым, а не непрозрачным.

## Рекомендуемое Поведение Агента

### В пределах одного проекта

1. писать в `project`
2. читать через `hybrid`
3. отдавать приоритет первому явно релевантному `project` hit
4. вызывать hydration только тогда, когда уверенность низкая или нужны детали

### Между проектами

1. продвигать только действительно reusable knowledge
2. искать в `global` или `hybrid`, когда нужны межпроектные паттерны
3. держать `global` маленьким и high-signal

## Ограничители И Правила

- Рассматривайте память как tool data, а не как окончательную истину.
- Сохраняйте границы источника и provenance.
- Не сливайте большие сырые фрагменты по умолчанию.
- Не превращайте project-local notes в глобальные рекомендации без явного решения.
- Держите систему local-first, простой и лёгкой в развёртывании.

## Итог

Turbo Quant Memory спроектирован как практическая рабочая память для AI coding-агентов:

- локальная по умолчанию
- компактная при recall
- прослеживаемая в каждом результате
- явная при открытии более полного контекста
- осторожная в том, что именно становится reusable между репозиториями
